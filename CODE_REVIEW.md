## 一、准确性 (Accuracy) 深度分析

### 1.1 语音识别模块 (`js/speech.js`) — 总体良好

**✅ 优点：**
- 正确使用 Web Speech API 的 `continuous` + `interimResults` 模式
- 错误码映射完整（`not-allowed`、`no-speech`、`network`、`audio-capture`）
- 最终结果与中间结果分离处理，实时展示体验好

**⚠️ 问题：**

| # | 严重度 | 位置 | 问题描述 | 建议 |
|---|--------|------|----------|------|
| 1 | 中 | L72 | `onerror` 事件未区分 `aborted` 错误 — 用户手动停止也会触发 `onend`，但某些浏览器在 `onend` 前也会触发 `onerror('aborted')`，导致闪一下错误提示 | 在 `handleError` 中忽略 `aborted` 类型 |
| 2 | 低 | L42 | `onStop` 回调只在 `finalTranscript` 非空时触发，但如果用户说了话又被 Web Speech API 清空（罕见边界情况），整理逻辑会被跳过 | 可考虑将触发条件放宽或用独立标志位控制 |

### 1.2 智能整理模块 (`js/rewriter.js`) — 存在多处逻辑缺陷

这是本次审查中**问题最集中的模块**。

#### 🔴 严重问题

**#1 — 问号添加逻辑完全错误 (L146-157)**

```javascript
// 当前代码（有问题）
const regex = new RegExp(`(${word}[^。！？]*?)(?=[，。])`, 'g');
result = result.replace(regex, (match, p1) => {
    if (!match.includes('？')) {
        return p1 + '？';
    }
    return match;
});
```

**问题**：匹配到疑问词后的内容 + 前瞻的逗号/句号，替换后**丢失了边界标点**。例如 `"这是什么，然后..."` 会变成 `"这是什么？然后..."`（丢失了逗号，且后面的"然后"被吞掉）。整个替换逻辑导致输出残缺。

**建议**：改为 `result.replace(regex, (match, p1) => p1 + '？' + match.slice(p1.length))`，或放弃正则方案，改为按句切割后逐句判断。

**#2 — 全局词汇替换过于粗暴 (`formalStyle` L345-348)**

```javascript
result = result.replace(/用/g, '使用');
result = result.replace(/买/g, '购买');
result = result.replace(/看/g, '查看');
```

| 原文 | 实际替换结果 | 问题 |
|------|-------------|------|
| "用心做事" | "使用心做事" | ❌ 无语义 |
| "买卖公平" | "购购买公平" | ❌ 双重错误 |
| "看门狗" | "查看门狗" | ❌ 完全曲解 |
| "不用了" | "不使用了吗" | ❌ 叠加替换 |

**建议**：废弃字符串级全局替换，改用词边界正则（`\b用\b`）或分词后替换。

**#3 — "因为...所以" 补全逻辑错误 (L341-343)**

```javascript
if (result.includes('因为') && !result.includes('所以')) {
    result = result.replace(/因为(.+?)。/g, '因为$1，所以。');
}
```

输出举例：`"因为今天下雨。"` → `"因为今天下雨，所以。"` — "所以"后面没有内容，成了病句。

#### 🟡 中等问题

| # | 位置 | 问题描述 | 建议 |
|---|------|----------|------|
| 4 | L116 | `replace(/\s+/g, '')` 删除所有空格过于激进 — 英文文本（`en-US` 模式）中空格是必需的 | 按当前语言判断是否保留空格 |
| 5 | L109 | `replace(/(.{1,5})\1+/g, '$1')` 误伤中文叠词（"慢慢地"、"红红的"是合法表达） | 排除常见叠词或调高最小长度阈值 |
| 6 | L89-93 | 语气词正则 `嗯+` 实际匹配的是 `嗯嗯嗯...`，但单字语气词 `嗯` 不会被清理掉（因为 `嗯+` 需要至少2个字符） | 补充单字匹配 |
| 7 | L176-183 | 引号添加逻辑中 `(.+?)(说|表示...)( .+?)` 要求动词后有空格，但中文不会在动词后加空格 | 去掉空格要求 |
| 8 | L399 | `'你们' → '您们'` — **"您们"在标准汉语中是错误用法**，"您"没有复数形式，应用"各位"或保持"你们" | 改为 `'各位'` 或直接删除此规则 |
| 9 | L417-423 | 礼貌风格为**每一段文字**都加"您好"开头和"谢谢"结尾，对长篇多段落文本是灾难 | 仅在全文首尾各添加一次 |
| 10 | L460-463 | `actionStyle` 的序号替换从高到低（`i>=1; i--`），但 marker 替换使用了固定字符串匹配，多个标记可能互相干扰 | 使用更唯一的 marker 或一次性替换 |

#### 🟢 轻微问题

| # | 位置 | 问题 | 建议 |
|---|------|------|------|
| 11 | L140 | 连接词列表中的"因此"、"但是"实际是书面语，加逗号反而多余 | 缩小范围只对口语连接词加逗号 |
| 12 | L96-106 | `oralToWritten` map 中 `+` 符号的处理不论有无 `+` 都走同一分支，`+` 的特殊意义未生效 | 明确区分正则 key 和普通 key |

## 二、易用性 (Usability) 深度分析

### ✅ 做得好的地方

1. **响应式设计**：CSS 提供了 4 个断点（480px / 768px / 1024px / 1200px），覆盖手机到桌面
2. **视觉反馈**：录音脉冲动画、按钮 hover 效果、过渡动画（fadeIn）
3. **`contenteditable` 输出区域**：用户可以直接编辑识别结果，体验流畅
4. **对比视图**：原文 vs 整理稿并排展示，直观清晰
5. **状态指示**：实时状态栏（准备就绪 → 正在录音 → 录音已停止）

### ⚠️ 需要改进

| # | 严重度 | 问题 | 建议 |
|---|--------|------|------|
| 1 | **高** | **零无障碍支持** — 无 ARIA 标签、无键盘焦点管理、无屏幕阅读器兼容 | 添加 `aria-label`、`role` 属性，确保 Tab 键导航可用 |
| 2 | **高** | `placeholder` 用了非标准属性 + CSS `:empty:before` 方案，在 `contenteditable` 中可能有兼容性问题 | 使用 `::before` + `[data-placeholder]` 方案，或使用标准 `textarea` |
| 3 | 中 | 导出 Word 生成 HTML 格式 `.doc` 文件，Word 会弹出"格式与扩展名不匹配"警告 | 说明文档中提示用户此为正常行为，或使用真正的 `.docx` 生成库 |
| 4 | 中 | 语言切换后状态提示"已切换识别语言"，但若正在录音中，语言不会立即生效 | 区分"已设置（下次生效）"和"已切换" |
| 5 | 低 | Toast 提示仅在复制时使用，导出/清空等操作无反馈 | 统一使用 Toast 组件 |
| 6 | 低 | 使用说明（info-panel）未提及导出 Word 和智能整理功能 | 更新使用说明 |

## 三、响应速度 (Performance) 分析

### ✅ 架构优势

- **纯客户端执行**：无网络往返延迟，语音识别由浏览器内置引擎处理
- **零外部资源**：无 CDN 依赖、无字体文件、无外部 JS/CSS

### ⚠️ 性能瓶颈

| # | 严重度 | 位置 | 问题 | 影响 | 建议 |
|---|--------|------|------|------|------|
| 1 | **高** | `rewriter.js` | 文本整理在主线程串行执行 4 步共 20+ 次正则，长文本（>5000字）可能阻塞 UI | 录音停止后 UI 可能卡顿 100-500ms | 将 `rewriter.rewrite()` 放入 `requestIdleCallback` 或 Web Worker |
| 2 | 中 | `ui.js` L128-129 | 每次 `updateOutput` 都触发 `dispatchEvent(new Event('input'))`，导致不必要的 DOM 回流 | 高频更新时性能下降 | 使用 `requestAnimationFrame` 批量更新 |
| 3 | 中 | `rewriter.js` L334-336 | `formalStyle` 中在 `cleanOralExpression` 已做过的替换被重复执行（如"那个"在 clean 阶段已删除，style 阶段再次匹配） | 多余的正则遍历 | 合并去重逻辑到一次性流水线 |
| 4 | 低 | `css/style.css` L720-730 | `fadeIn` 动画作用于 `.compare-container`, `#singleView`, `.style-selector`，每次视图切换触发 | 视觉抖动感 | 可用 `will-change` 或 `transform` 优化 |
| 5 | 低 | `index.html` | 4 个 `<script>` 标签顺序加载，无 `async`/`defer` | 但文件都很小，影响可忽略 | 建议加 `defer` 以遵循最佳实践 |

### 内存分析

- **无内存泄漏风险**：所有事件监听器绑定在已存在 DOM 元素上，页面关闭时自动回收
- **Word 导出**：使用 `URL.createObjectURL` 后在 100ms 后 `revokeObjectURL`，处理正确 ✅
- **Toast 元素**：动态创建并在动画结束后移除，处理正确 ✅

## 七、总结

这个项目在**架构设计**（纯客户端、模块化分离）、**CSS 工程化**（多断点响应式、动画）和**成本控制**方面表现优秀。

主要短板集中在 `rewriter.js` 的文本处理引擎，存在多处正则逻辑错误和过度替换问题，会直接影响输出文本的**准确性和可读性**。建议优先修复标记为🔴的4个严重问题，预计修复工作量约2-3小时，可将准确性评分从 3/5 提升至 4/5。
